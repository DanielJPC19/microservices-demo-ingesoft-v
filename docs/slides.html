¿Qué construimos?

Sistema de votación en tiempo real con arquitectura de microservicios polyglot, despliegado en Kubernetes sobre GCP.

  • Vote Service — Java / Spring Boot · recibe votos
  • Worker Service — Go · procesa y persiste
  • Result Service — Node.js · muestra resultados live
  • Kafka — broker de mensajes (KRaft)
  • PostgreSQL — base de datos de resultados

Patrón central: CQRS

Ruta de escritura (Command) completamente separada de la ruta de lectura (Query)

Estrategia DevOps: Two-Repo

Repositorio de desarrollo (CI) separado del repositorio de operaciones (CD + Terraform)

Arquitectura CQRS

Write Path — Command

👤 Usuario HTTP POST /vote
→
☕ Vote Service Java · Spring Boot
→
📨 Kafka topic: votes
→
⚙️ Worker Go · consumer
→
🐘 PostgreSQL INSERT / UPSERT

Read Path — Query

🐘 PostgreSQL SELECT COUNT
→
📊 Result Service Node.js · polling 1s
→
👤 Usuario Socket.io live

Los dos servicios de cara al usuario (Vote y Result) exponen IPs externas vía LoadBalancer. El Worker no tiene puerto externo. Las credenciales de BD viajan como Kubernetes Secret.

Patrones de Nube Implementados

⚡

Rate Limiter

Vote Service

Máximo 5 requests/segundo por instancia via Resilience4j. Si se supera, respuesta inmediata con mensaje de error sin lanzar excepción.

🔄

Retry + Backoff

Worker Service

Exponential backoff con cenkalti/backoff/v4 en conexión a PostgreSQL y Kafka (500ms → 30s). En escrituras: 3 reintentos con intervalo de 1s.

🔌

Circuit Breaker

Kafka

Kafka actúa como buffer asíncrono: si el Worker cae, los votos se acumulan en el topic sin que Vote falle. Rompe la cadena de fallos en cascada.

🧱

Bulkhead

GKE Pods

Cada microservicio corre en su propio pod aislado con límites de CPU y memoria. Un pod saturado no afecta a los demás.

¿Por qué kubectl en vez de Helm Provider?

El Terraform Helm provider tenía incompatibilidad con clusters GKE creados en el mismo apply. Error: context deadline exceeded a los 15 min. Solución: imágenes Docker directas (postgres:16, apache/kafka:3.7.0) vía kubectl.

Workload Identity Federation

GitHub Actions se autentica en GCP via OIDC, sin almacenar Service Account keys. La condición de atributo limita el acceso al repositorio exacto de ops.

Errores Clave & Soluciones

Error Causa Solución
Helm timeout 5 min listeners.interbroker.protocol es solo ZooKeeper — crashea Kafka KRaft Eliminar ese parámetro del chart
Helm timeout 15 min Incompatibilidad del Terraform Helm Provider con GKE dinámico Reemplazar Helm por kubectl apply directo con imágenes Docker
409 Already Exists Cluster existía en GCP pero no en el state de Terraform (GCS) terraform import o limpiar state + recrear
WIF 403 Faltaba attribute-condition en el provider OIDC Agregar condición assertion.repository == 'org/repo-ops'
HCL syntax error Punto y coma (;) en variables HCL — inválido Formato multilínea en todos los bloques
Git push 403 Token sin scope repo, sin terminal para credenciales Clave SSH ed25519 + ~/.ssh/config + remote SSH
