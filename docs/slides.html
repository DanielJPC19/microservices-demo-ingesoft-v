¿Qué construimos?

Sistema de votación en tiempo real con arquitectura de microservicios polyglot, desplegado en Kubernetes sobre GCP.

  • Vote Service — Java / Spring Boot · recibe votos
  • Worker Service — Go · procesa y persiste
  • Result Service — Node.js · resultados live
  • Kafka — broker de mensajes (KRaft)
  • PostgreSQL — base de datos

Patrón central: CQRS

Ruta de escritura (Command) completamente separada de la ruta de lectura (Query)

Estrategia DevOps: Two-Repo

Repositorio de desarrollo (CI) separado del repositorio de operaciones (CD + Terraform)

Patrones de Nube Implementados

⚡

Rate Limiter

Vote Service

Máximo 5 requests/segundo via Resilience4j. Rechaza inmediatamente con fallback sin lanzar excepción.

🔄

Retry + Backoff

Worker

Exponential backoff con cenkalti/backoff/v4. Conexión a PostgreSQL y Kafka: 500ms → 30s. Escrituras: 3 reintentos × 1s.

🔌

Circuit Breaker

Kafka

Kafka como buffer asíncrono: si el Worker cae, los votos se acumulan en el topic sin que Vote falle — rompe la cascada.

🧱

Bulkhead

GKE Pods

Cada microservicio en su pod aislado con límites de CPU y memoria. Un pod saturado no afecta a los demás.

@PostMapping("/")
@RateLimiter(
  name = "voteRateLimiter",
  fallbackMethod = "rateLimitFallback"
)
String postForm(
    @CookieValue String voterId,
    @ModelAttribute Vote voteInput,
    Model model, ...) {
  // publica en Kafka
  kafkaTemplate.send(KAFKA_TOPIC, voter, vote);
  return "index";
}

// Invocado automáticamente si se supera el límite
String rateLimitFallback(...,
    RequestNotPermitted ex) {
  logger.warn("Rate limit exceeded: {}", voterId);
  model.addAttribute("error",
    "Demasiadas solicitudes. Espera un momento.");
  return "index";
}

Retry + Backoff — Implementación

Worker Service · Go
Go — pingDatabase()Conexión con exponential backoff
func pingDatabase(db *sql.DB) {
  bo := backoff.NewExponentialBackOff()
  bo.InitialInterval = 500 * time.Millisecond
  bo.MaxInterval     = 30  * time.Second

  // Reintentos: 500ms → 1s → 2s → ... → 30s
  err := backoff.RetryNotify(
    db.Ping,
    bo,
    func(err error, d time.Duration) {
      log.Printf(
        "DB ping failed, retrying in %s: %v",
        d, err)
    },
  )
  if err != nil {
    log.Panic("Could not ping database:", err)
  }
}
Go — escritura con retry3 reintentos × 1s al escribir
// Por cada mensaje Kafka recibido:
insertStmt := `insert into "votes"("id","vote")

writeOp := func() error {

    string(msg.Key),
    string(msg.Value))
  return err
}

bo := backoff.WithMaxRetries(
  backoff.NewConstantBackOff(time.Second),
  3,
)
backoff.RetryNotify(writeOp, bo,
  func(err error, d time.Duration) {
    log.Printf("DB write retry: %v", err)
  })

kubectl get pods/services

¿Por qué kubectl en vez de Helm Provider?

El Terraform Helm Provider tenía incompatibilidad con clusters GKE creados en el mismo apply. Error: context deadline exceeded a los 15 min. Solución: imágenes Docker directas (postgres:16, apache/kafka:3.7.0) vía kubectl.

Workload Identity Federation

GitHub Actions se autentica en GCP via OIDC sin almacenar Service Account keys. La condición de atributo limita el acceso al repositorio exacto de ops.

resource "google_container_cluster" "main" {
  name     = "votes-prod"
  location = var.region

  # Elimina el node pool por defecto
  # y usa el nuestro (más configurable)
  remove_default_node_pool = true
  initial_node_count       = 1
  deletion_protection      = false
}

resource "google_container_node_pool" "nodes" {
  name       = "default"
  location   = var.region
  cluster    = google_container_cluster.main.name
  node_count = 2

  node_config {
    machine_type = "e2-medium"
    disk_size_gb = 30
    oauth_scopes = ["https://www.googleapis.com/auth/cloud-platform"]
  }
}

Pipeline CD — cd-infra.yml

ops repo · workflow_dispatch
YAML — Autenticación + TerraformWIF sin JSON keys
- name: Authenticate to GCP (WIF)
  uses: google-github-actions/auth@v2
  with:
    workload_identity_provider:
      ${{ secrets.WIF_PROVIDER }}
    service_account:
      ${{ secrets.WIF_SA }}

- name: Terraform Apply (GKE only)
  working-directory:
    infra/envs/${{ env.ENV }}
  run: |
    terraform init
    terraform apply -auto-approve \
      -var="image_tag=${{ env.IMAGE_TAG }}"
  env:
    TF_VAR_project_id:
      ${{ secrets.GCP_PROJECT_ID }}
    TF_VAR_db_password:
      ${{ secrets.DB_PASSWORD }}
YAML — kubectl applyApps vía kubectl directo
- name: Get GKE credentials
  run: |
    gcloud container clusters \
      get-credentials votes-${{ env.ENV }} \
      --region us-central1

- name: Create DB secret
  run: |
    kubectl create secret generic db-secret \
      --from-literal=password=${{ secrets.DB_PASSWORD }} \
      --dry-run=client -o yaml | kubectl apply -f -

- name: Deploy Vote, Worker, Result
  env:
    DOCKER_USERNAME: ${{ secrets.DOCKER_USERNAME }}
  run: |
    kubectl apply -f - <<EOF
    # image: ${DOCKER_USERNAME}/vote:${IMAGE_TAG}
    # image: ${DOCKER_USERNAME}/worker:${IMAGE_TAG}
    # image: ${DOCKER_USERNAME}/result:${IMAGE_TAG}
    EOF

Helm timeout 15 min Incompatibilidad del Terraform Helm Provider con GKE dinámico en el mismo apply Reemplazar Helm por kubectl apply con imágenes Docker directas
409 Already Exists Cluster existía en GCP pero no en el state GCS (state vacío) terraform import o limpiar state + recrear cluster
NodePool ERROR Quota de CPU agotada (8 vCPU límite · staging corriendo) Reducir node_count 3→2 en prod · destruir staging primero
WIF 403 Faltaba attribute-condition en el OIDC provider Agregar condición assertion.repository == 'org/repo-ops'
